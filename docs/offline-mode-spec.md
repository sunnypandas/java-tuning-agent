# 离线分析模式（生产导出 → 测试/本机分析）需求说明

## 1. 背景与目标

- 当前能力以**本机发现 Java 进程**并连接 **JVM 调优/诊断流水线**为主；生产环境常需先导出 dump、日志等，再在隔离环境分析。
- 本需求增加**离线/导入分析模式**：不依赖目标机器上可 `jcmd` 的 live 进程，而是基于用户提供的**文件与元数据**完成等效或降级的分析。
- **目标效果**：
  - 用户按向导**分步**补全输入；**参数不足时允许降级继续**，并在输出中明确缺失与置信度影响。
  - **与现有在线模式兼容**：不传离线相关参数时，现有 Discover → Inspect → Scope gate → Evidence → Advice 行为**不改变**。
  - 大体量文件（尤其是 heap dump）支持**分块传输与校验**。

---

## 2. 术语

| 术语 | 含义 |
|------|------|
| **在线模式** | 沿用现有链路：发现本机 JVM、`inspectJvmRuntime`、`collectMemoryGcEvidence`、`generateTuningAdvice`（含 step-3 gate）。 |
| **离线模式 / 独立模式** | 不依赖 live PID 采集的新分析路径；输入来自用户导入/粘贴的导出包。 |
| **必选输入** | normative 收集目标；应尽量齐全。**不齐时允许降级**，不得在向导层硬性阻断全流程（见 §6）。 |
| **推荐输入** | 显著提升结论质量；缺失时须有**显式「本次没有」**选项并记入报告。 |
| **背景信息** | 辅助解释环境与上下文；不参与硬性校验。 |

## 3. 输入分层：必选、推荐、背景

以下与「手工导出清单」对齐；离线向导的收集顺序可在此基础上编排（见 §6）。

### 3.1 必选（目标齐全；不齐则降级）

| ID | 内容 | 说明 |
|----|------|------|
| B1 | JVM 与进程标识 | 完整命令行（含全部 JVM 参数）、PID、采集时间戳（文本即可）。 |
| B2 | JDK 基本信息 | 版本、发行商等（与线上一致或可比对）。 |
| B3 | 只读运行时快照 | 与线上一键 inspect **等价**的 jcmd/jstat 类**文本**（堆/GC 摘要等）。 |
| B4 | 类直方图 | `GC.class_histogram` 等输出的文本文件或粘贴内容。 |
| B5 | 线程 dump | `Thread.print` / `jstack` 等文本。 |
| B6 | 堆转储 | `.hprof`；与同批直方图、线程 dump **同一时间窗口**为佳。 |

**策略说明**：为简化业务规则，**不要求**用户按问题类型三选一；三类深入证据（B4–B6）作为**统一必选目标**一并收集。若实际交付缺失其中若干项，进入 **§6 降级策略**，不得在交互上无限阻塞。

### 3.2 推荐（显式缺失）

| ID | 内容 | 说明 |
|----|------|------|
| R1 | GC 日志 | 与问题时段重叠的 GC 日志（如 `-Xlog:gc*` 或历史格式）。 |
| R2 | 应用日志 | 同一时段内与异常、超时、OOM、发布相关片段。 |
| R3 | 多次轻量采样 | 同窗口 2～3 次 `jstat` 或间隔 thread/histogram，用于区分瞬态与趋势。 |

每一项须支持用户**显式选择「本次没有」**，系统记录该选择用于报告中的「缺失数据」章节，而非静默跳过。

### 3.3 背景信息（可选参考）

- OS / 容器资源约束（内存上限、cgroup、`ulimit` 等）。
- 发布与配置上下文（版本、镜像 tag、启动脚本快照）。
- 业务与负载说明、监控摘要、工单摘要等非文件信息。

## 4. 独立模式与在线模式的边界

| 维度 | 要求 |
|------|------|
| 入口 | **独立模式**为单独入口（具体工具名/命令/触发方式在实现时确定，本文不锁死）。 |
| 兼容性 | **默认**未启用离线参数时，**仅**走现有在线流水线；不得破坏既有 MCP 契约与调用方。 |
| 会话 | **一次对话**内完成整套离线 input（含补全、回退）；**不做**跨会话持久化状态（与当前约定一致）。 |
| 语言 | 向导与错误提示**默认中文**。 |

---

## 5. 新增工具与 Consent

- 离线分析通过**新增专用工具**（或协作的一组工具）实现输入收集与分析触发，**不替换**现有 `listJavaApps` / `inspectJvmRuntime` / `collectMemoryGcEvidence` / `generateTuningAdvice` 的语义。
- **Consent 语义与线上一致**：继续使用同一套 `confirmationToken`（及 heap 路径等既有规则中与授权相关的字段）表达用户同意；离线场景下应对齐为「用户确认使用本批导入材料进行相应分析」，**不**另起一套审批模型，以免调用方混乱。

## 6. 向导交互需求

### 6.1 顺序与方式

- **固定顺序**的分步向导（顺序表在实现时可映射为状态机；步骤与 §3 各条对应）。
- 每一步同时支持：
  - **本地文件路径**（或工作区可读路径）；
  - **粘贴文本**（与文件二选一或补充，具体以工具参数设计为准）。
- **允许回退**：用户可回到前面某一步**仅修改该步**输入，不必清空后续已填项（实现上可采用「从第 N 步重填」或显式步骤号，需在实现方案中写明）。

### 6.2 输入不足与降级

- **不**因必选层缺项而**硬性拦截**整条分析链路；在缺项情况下**允许用户选择继续**，进入分析阶段。
- 输出侧必须：
  - 列出**缺失的必选/推荐项**；
  - 说明对**结论与置信度**的影响（可与现有 `formattedSummary` 中 Missing data / Confidence 等章节对齐）。

### 6.3 大文件：分块与校验

- 对 **`.hprof` 等大体量文件**须支持：
  - **分块**上传或分块写入的等价机制；
  - 合并后的**完整性校验**（如哈希、大小一致）；
  - 校验失败时：提示错误并允许**仅重试该步**。
- 分块大小、哈希算法、重试次数等**实现参数**可在开发阶段定义，本文只要求能力与用户体验。

### 6.4 推荐项「本次没有」

- 对 R1–R3 类推荐输入，向导须提供明确 UI/参数：如 **「本次没有 GC 日志」**，并写入结构化元数据供报告使用。

---

## 7. 与非功能需求的提示

- **安全**：本文不规定脱敏、加密与留存策略；上线前若处理含 PII 的 dump/日志，应另补安全需求。
- **性能**：离线 heap 分析可能占用大量内存与磁盘；可在实现说明中给出建议上限或流式策略（待定）。

---

## 8. 验收要点（草案）

- [ ] 仅使用旧参数调用时，在线模式行为与当前版本一致（回归）。
- [ ] 离线模式可从入口走完向导；必选项缺失时可继续并得到带「缺失与降级」说明的报告。
- [ ] 推荐项可选「本次没有」，报告中可见对应缺失说明。
- [ ] `.hprof` 支持分块与校验失败重试。
- [ ] 向导支持回退修改某步；中文默认文案可用。
- [ ] 新增工具仍使用与线上一致的 consent/token 语义（具体字段名以 MCP schema 为准）。
